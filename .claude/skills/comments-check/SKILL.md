# 注释检查 — 完整性 · 准确性 · 可读性

请按以下流程逐步检查当前项目的代码注释质量，最后给出一份中文报告。

---

## 检查标准

本项目最佳实践：
- Python 代码注释用中文，关键函数写 docstring（`"""..."""`）
- TypeScript/React 代码注释用中文，关键组件写 JSDoc
- 目标：**关键函数和复杂逻辑必须有注释**，不强制行数比例，但核心模块注释覆盖率高

---

## 第 1 步：确定检查范围

先问用户要检查哪些文件。如果用户没指定，默认检查最近修改过的源码文件（通过 `git diff --name-only HEAD~1` 或 `git status` 查看，若无 git 则检查最近修改时间的文件）。

只检查源码文件：`.py`、`.ts`、`.tsx`、`.js` 等，跳过 `node_modules/`、`__pycache__/`、`package-lock.json`、`.min.js`、`backend/data/` 等。

---

## 第 2 步：逐文件检查

对每个文件，从三个维度检查：

### 维度 A：注释覆盖率（完整性）

对 Python 文件：
- 每个函数/方法是否有 docstring？
- 类定义是否有说明注释？
- 关键逻辑（条件判断、异常处理、数据处理管道）是否有行内注释？
- 统计注释行数（`#` 和 `"""..."""` 的行）

对 TypeScript/React 文件：
- 每个组件是否有用途说明？
- 关键 hooks（useEffect、useCallback、useState 复杂初始化）是否有注释？
- 复杂 props 接口是否有字段说明？
- 统计注释行数（`//` 和 `/* */` 的行）

打分：⭐（严重不足）← → ⭐⭐⭐⭐⭐（完全达标）

### 维度 B：注释准确性

对每个函数/方法的注释，读注释 → 读代码 → 对比：
- 注释说的功能，代码真的实现了吗？
- 有没有"过期注释"（代码已改但注释没改）？
- 注释里的参数名、返回值描述是否和代码一致？
- Python docstring 中的 `:param` / `:return` 是否与实际签名匹配？

打分：⭐（多处不匹配）← → ⭐⭐⭐⭐⭐（完全匹配）

### 维度 C：小白可读性

用"完全不懂编程的人"的视角读每条注释：
- 注释用了专业术语但没解释？（如"RAG管道""SSE流式""向量嵌入""JWT Token"）
- 注释是写给"未来的自己"还是"任何一个接手的人"？
- 注释能不能让读者在不看代码的情况下，理解这段代码要干什么？

打分：⭐（全是术语）← → ⭐⭐⭐⭐⭐（小白友好）

---

## 第 3 步：生成报告

用以下格式输出：

```
📋 注释检查报告
══════════════════════════════════
检查时间：YYYY-MM-DD HH:MM
检查范围：X 个文件，共 Y 个函数/方法
══════════════════════════════════

📊 总体评分
┌──────────────┬───────┐
│ 完整性       │ ⭐⭐⭐  │
│ 准确性       │ ⭐⭐⭐⭐ │
│ 小白可读性    │ ⭐⭐    │
│ 综合         │ ⭐⭐⭐  │
└──────────────┴───────┘

────────────────────────────────

📁 文件：backend/app/rag/pipeline.py
  函数/方法数：8 个，已注释：5 个

  ❌ 缺少注释的函数：
    - estimate_tokens()  → 建议：加 docstring "粗略估算文本的 token 数量"
    - _extract_citations() → 建议：加 docstring "从检索文档中提取引用信息"

  ⚠️ 注释不匹配：
    - line 68 _build_chat_messages: 注释说"构建多轮对话消息列表"，实际还包含了系统提示词组装

  💡 小白友好度低：
    - line 63 "Chroma" → 建议加注释"Chroma 是本地向量数据库，用于存储和检索文档的语义向量"

────────────────────────────────

📁 文件：frontend/src/pages/ChatPage.tsx
  ...

══════════════════════════════════
🔧 修复建议（按优先级）
══════════════════════════════════
1. [高] xxx 函数缺少 docstring/注释
2. [中] xxx 注释用词过于专业
3. [低] xxx 可补充行内注释
```

---

## 第 4 步：询问是否修复

报告生成后，问用户：
- "要我帮你补充缺失的注释吗？"
- "要我修正不准确的注释吗？"

等用户确认后再动手修改代码。
